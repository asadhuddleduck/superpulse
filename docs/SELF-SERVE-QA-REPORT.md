# Self-serve onboarding — deep regression QA report

Generated 30 Jun 2026 by a multi-Opus adversarial QA swarm (6 dimension finders → 3 skeptics per finding, ≥2/3 to survive → fixes → tsc repair). Follow-on to commit `f094ccb`.

**Result:** 25 confirmed findings, 9 refuted. All confirmed findings fixed (2 marked no-change-needed as duplicates of the re-OAuth fix). `tsc --noEmit` + `npm run build` green.

## ⚠️ Required before deploy
- **Turso migration (additive):** `ALTER TABLE tenants ADD COLUMN provisioning_failed_reason TEXT;` and `ALTER TABLE tenants ADD COLUMN needs_reauth INTEGER DEFAULT 0;` on the `superpulse` DB. New code reads these; deploying before the migration will error.
- **Forced re-login:** the `tenant_id` cookie is now HMAC-signed, so every currently-logged-in user (incl. legacy clients + Asad) is logged out once and must re-connect via Facebook. Expected, one-time.
- **`HQ_SIGNING_SECRET`** must be set in every env (prod ✓). The signer fails closed without it.

## 🔴 Critical

### [fixed] tenant_id session cookie is unsigned plaintext — forging it is full takeover of any tenant (incl. decrypted Meta token + ad spend)
`src/lib/auth.ts:50` · security

**Failure:** The entire customer app authenticates a user solely by the value of the httpOnly `tenant_id` cookie: setTenantCookie (line 17-26) stores the bare tenant id as the cookie value, and getCurrentTenant/getTenantCookie (line 50-59) just call getTenantById(<cookie value>) with no HMAC, signature, or identity re-check (grep for hmac/sign/verify over tenant_id returns nothing). httpOnly only blocks JS, not the user's own devtools/proxy. An attacker who sets `tenant_id=<victim tenant id>` (e.g. a legacy paying client `t_<igUserId>` or `cust_<stripeCustomerId>`) is then treated as that tenant everywhere: GET /api/locations returns the victim's locations; /dashboard renders the victim's account (the layout at dashboard/layout.tsx:76 even calls fetchMe with the VICTIM's stored decrypted token, so the validity check passes and confirms the impersonation rather than blocking it); POST /api/onboarding/budget and /api/onboarding/select-ad-account mutate the victim's account and spend on the victim's ad account; getCurrentToken() drives Meta calls with the victim's decrypted token. No password or token is ever required.

**Fix:** Stop trusting a plaintext identifier as the credential. Either (a) store a random opaque session token in the cookie and map it server-side to a tenant id (exactly the pattern hq-auth.ts already uses for sp_hq: randomToken + sha256 at rest + expiry/revocation), or (b) HMAC-sign the cookie value `tenantId.<hmac(secret,tenantId)>` and verify+timingSafeEqual in getTenantCookie before resolving. Bind issuance to the authenticated Facebook identity so a cookie can't name a tenant the holder never authenticated as.

### [fixed] £90 onboarding handhold has no durable record — relies solely on fire-and-forget notifySlack, which Vercel can drop after the response
`src/app/api/webhook/stripe/route.ts:266` · payments · regression from f094ccb

**Failure:** The new handhold branch in handleAuditPayment writes NOTHING to the DB and its entire effect is `void notifySlack(...)` (not awaited) before returning. After the route returns its 200 NextResponse, the serverless invocation can be frozen/killed before the un-awaited fetch in notifySlack (slack.ts:13-36) completes, or the Slack POST simply fails (its catch swallows the error). Result: customer paid £90, there is no audit_purchases row, no DB record, and no alert anywhere -> Asad never learns to call them and the customer is stranded. Conversely, because there is also no idempotency guard, a genuine Stripe at-least-once redelivery of checkout.session.completed re-fires the same '£90 onboarding handhold purchased' Slack message, so Asad may action/call the same buyer twice.

**Fix:** Persist the handhold to a durable table (or reuse audit_purchases with tier='handhold') via INSERT ... ON CONFLICT(stripe_session_id) DO NOTHING, and only fire the Slack when rowsAffected>0 (dedup). Await the notifySlack (or use Next.js after()/waitUntil) so a failed alert surfaces as a non-2xx and Stripe retries, instead of being silently lost.

### [fixed] C1: chk: checkout session is reusable and not identity-bound — one subscription can be attached to multiple IG tenants, and re-running OAuth with a different IG hijacks the sub and bricks the original tenant
`src/app/api/auth/callback/facebook/route.ts:262` · oauth · regression from f094ccb

**Failure:** applyCheckoutSubscription (def L147-177) trusts checkoutSessionId straight out of the OAuth `state` (chk:<id>, derived from the public /onboarding/connect?session_id= URL) and never (a) checks session.customer_details.email == the authenticated FB user, (b) marks the session consumed, or (c) restricts the placeholder merge. Concrete: buyer pays once (cust_C, 1 location) and connects IG-A → t_<IG-A> gets the sub, cust_C merged (correct). The buyer (or anyone who has the link from browser history / the Stripe success_url / referrer) revisits /onboarding/connect?session_id=X and logs in with a DIFFERENT IG-B they own. The callback retrieves session X again (Stripe sessions stay retrievable), stamps cust_C + the same subscription + paid_locations onto t_<IG-B>, then calls neutraliseCustomerPlaceholder(t_<IG-B>, C). That UPDATE (tenants.ts:350-354) matches `stripe_customer_id = C AND id != t_<IG-B>` with NO placeholder restriction, so it hits the fully-onboarded t_<IG-A>: sets it status='merged', subscription_status='merged', stripe_customer_id=NULL. Result: the original paying tenant is silently dropped from getActiveTenants/getTenantsAwaitingProvision (status no longer 'active'), its ads management halts, while t_<IG-B> now runs a second tenant on one £27 subscription. Before f094ccb the single-Page branch never called applyCheckoutSubscription, so this neutralise/hijack path was unreachable on the dominant one-Page flow; C1 puts it live there.

**Fix:** Bind + consume the session: (1) require session.payment_status==='paid' && session.status==='complete'; (2) verify the authenticated user's email (fetchMe email / session.customer_details.email) matches before stamping; (3) record consumed checkout session ids (e.g. a stripe_checkout_session_id column or a consumed-sessions table) and no-op if already attached to a different tenant; (4) narrow neutraliseCustomerPlaceholder's WHERE to genuine placeholders only (e.g. `AND id LIKE 'cust\_%' AND status='pending_oauth'`) so it can never merge a real IG tenant.

### [fixed] C1 fix has no durability: a transient Stripe error during the OAuth callback permanently strands the paid customer on /pricing — the exact symptom C1 set out to fix
`src/app/api/auth/callback/facebook/route.ts:174` · oauth · regression from f094ccb

**Failure:** applyCheckoutSubscription wraps everything in try/catch and only console.error()s on failure (L174-176); it is the SOLE path that links a checkout subscription to an IG-keyed tenant (confirmed: only caller is this callback; the Stripe webhook keys exclusively on stripe_customer_id, which the IG tenant will NOT have if the link step failed; src/app/api/cron/reconcile has no billing reconcile). If stripe.checkout.sessions.retrieve (L150) throws for any reason at that one moment — Stripe rate-limit/5xx/network blip — the subscription is never attached, subscription_status stays 'pending', and there is no retry and no background reconcile. The webhook (which created cust_<customerId> with subscription_status='active') cannot find the IG tenant because the IG tenant's stripe_customer_id is still NULL. The user is logged into t_<igUserId> (cookie set), so the dashboard gate sees subscription_status='pending' and redirects to /pricing — money orphaned on the cust_ placeholder, identical to the pre-C1 bug. The 'fix' is a single best-effort call with no idempotent retry.

**Fix:** Make the link durable: persist the checkout session id on the tenant at connect time and add a reconcile pass (cron or webhook-side) that maps cust_ placeholders / pending IG tenants to their checkout session and re-attaches; OR retry the retrieve with backoff and, on hard failure, store the pending session_id so a later page-load/cron completes the attach. At minimum, fall back to the webhook-created cust_ tenant by email so the paid state is recoverable without the live Stripe call succeeding.

### [fixed] applyCheckoutSubscription writes the RAW Stripe subscription status (and defaults to 'active' when no subscription is present), bypassing the webhook's statusMap — non-active subs pass the dashboard billing gate
`src/app/api/auth/callback/facebook/route.ts:159` · oauth · regression from f094ccb

**Failure:** L159: `const status = typeof sub === "object" && sub?.status ? sub.status : "active"` then setTenantStripeFields writes it raw (L160-164). The Stripe webhook normalizes statuses via statusMap (route.ts:164-173: incomplete→pending, unpaid→past_due, incomplete_expired→canceled), but applyCheckoutSubscription does NOT. The dashboard gate (dashboard/layout.tsx:27-37) only redirects to /pricing for falsy/'pending'/'canceled'/'past_due'. So: (a) a buyer whose card needs 3DS/SCA reaches success_url with the subscription in Stripe status 'incomplete' — applyCheckoutSubscription writes subscription_status='incomplete', which is not in the redirect set, so the dashboard renders for an unpaid subscription. (b) If the session has a customer but session.subscription is null (expired/open session id passed manually to /onboarding/connect), the fallback writes subscription_status='active' with no subscription on file, granting active billing state for free. Both bypass the intended gate; only later webhooks may correct it, and only if customer mapping survives.

**Fix:** Run the Stripe status through the same statusMap the webhook uses (single shared helper), and refuse to write 'active' unless a subscription id exists and Stripe reports active/trialing; treat absent/incomplete/unpaid as 'pending' so the gate behaves identically on both write paths.

### [no_change_needed] C2/C3: provision_failed for a Meta-access reason is a permanent dead-end — re-auth (the remedy in the Slack alert) never resets provisioning_status, and nothing re-enters the cron
`src/app/api/cron/v8/provision/route.ts:116` · oauth · regression from f094ccb

**Failure:** C2 parks tenants in provision_failed for meta_token_invalid / meta_permission_denied (L116). The ONLY transition back to 'provisioning' anywhere in the codebase is setTenantBudget, called solely from /api/onboarding/budget (verified by grep: setProvisioningStatus only writes pending_locations/pending_budget/provisioning_failed/provisioned; 'provisioning' is only set by setTenantBudget). So after the most common access failure — the 60-day long-lived token expiring → meta_token_invalid — the user re-auths via Facebook (exactly what the Slack ping recommends), but the OAuth callback's upsertTenant only updates status/token and never touches provisioning_status (it stays provision_failed). The dashboard sees provision_failed + a valid budget → renders the dashboard with no redirect (layout.tsx:65-71). getTenantsAwaitingProvision excludes provision_failed, so the cron never picks them up again. Net: token is fresh, access is fixed, yet provisioning is permanently stalled with no UI affordance to resume — only a manual DB edit or a coincidental budget re-approval recovers it.

**Fix:** On a successful re-auth/OAuth for a tenant currently in provision_failed (or when ad-account/token is refreshed), reset provisioning_status back to 'provisioning' so the cron retries; and/or add an explicit 'Retry provisioning' action on the dashboard for access-cause failures that flips provision_failed→provisioning.

### [no_change_needed] Access-reason provision_failed never resumes after the human fix — C2's 'stuck forever' bug reintroduced for the access branch
`src/app/api/cron/v8/provision/route.ts:116` · statemachine · regression from f094ccb

**Failure:** Tenant's campaign-create fails with an access/token error (expired token, ads_management not granted, not yet an app tester). classifyMetaAccessError matches → setProvisioningStatus(tenant.id,'provision_failed') + Slack ping. provision_failed is EXCLUDED from getTenantsAwaitingProvision (tenants.ts:198 only selects 'provisioning'/'provisioned'). The dashboard layout sees budget VALID and renders the dashboard (no redirect). Asad then does the Slack-recommended fix (re-auth / adds the user as an app tester / restores ad-account access). ### [fixed] classifyMetaAccessError misses Meta error code 294 (and 100/subcode-33 'missing permissions') — the exact 'no ads_management grant / lost ad-account access' case C2 was built for loops forever
`src/lib/meta-errors.ts:157` · v8engine · regression from f094ccb

### [fixed] One paid Stripe checkout session can be replayed onto unlimited tenants — and a replay clobbers the real buyer's tenant to status='merged'
`src/app/api/auth/callback/facebook/route.ts:262` · security · regression from f094ccb

**Failure:** applyCheckoutSubscription (147-177) treats the Stripe checkout session id carried in the OAuth `state` (`chk:<id>`, set in onboarding/connect/page.tsx:34) as a bearer credential. It retrieves the session and stamps stripe_customer_id / stripe_subscription_id / subscription_status='active' / paid_locations=qty onto whatever IG-keyed tenant the OAuth resolves to, with NO check that the session was already consumed (confirmed: no single-use table/column) and NO binding to the authenticated identity (the in-code comment claims the id 'doubles as the CSRF nonce' but it is never compared to any cookie). f094ccb added this exact call to the single-Page branch (line 262) — per the commit message the dominant local-business path — so the dominant funnel is now exposed. Concrete abuse: pay once for qty=1 (£27), then revisit `/onboarding/connect?session_id=cs_live_X` and complete OAuth with N different Instagram/Facebook accounts; each resolves to a distinct tenant `t_<igUserId>` and each is marked active + paid + provisioned (real ads created on Meta) off the single subscription. Worse, the second bind corrupts the first: neutraliseCustomerPlaceholder(tenant2, cus_X) runs `UPDATE tenants SET status='merged', subscription_status='merged', stripe_customer_id=NULL WHERE stripe_customer_id='cus_X' AND id != tenant2` (tenants.ts:346-356), which matches the legitimate buyer's tenant1 (it shares cus_X) and locks the paying customer out of their dashboard — cross-tenant account hijack + denial of service from a leaked/shared session id.

**Fix:** Make the session single-use and identity-bound: persist consumed checkout-session ids (unique constraint) and refuse re-application; verify the session's customer/subscription isn't already attached to a different live tenant before stamping; scope neutraliseCustomerPlaceholder to placeholder rows only (e.g. AND status NOT IN ('active','provisioned') or only id LIKE 'cust_%') so it can never merge a real IG tenant; add a real OAuth state nonce stored in an httpOnly cookie (as the gate flow does) instead of relying on the session id as the nonce.
